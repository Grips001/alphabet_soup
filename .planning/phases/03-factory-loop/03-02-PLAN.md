<task type="auto" tdd="true">
  <name>Task 1: InserterSystem with smart filter</name>
  <files>
    src/game/systems/InserterSystem.ts,
    src/game/systems/InserterSystem.test.ts
  </files>
  <behavior>
    Basic transfer:
    - Inserter with belt at pickup tile containing a LetterItem: after INSERTER_TRANSFER_INTERVAL ticks, item moves to inserter.heldItem, belt.item becomes null
    - Next transfer cycle: if dropoff belt is empty, heldItem moves to belt.item, inserter.heldItem becomes null
    - If dropoff belt is occupied, inserter keeps holding item, retries next cycle
    - ticksSinceLastTransfer resets to 0 after successful pickup or deposit

    Smart filter (assembler target):
    - Inserter facing an assembler with recipe "SET": only picks up S, E, T from belt; ignores other letters
    - Inserter facing an assembler with recipe "SEE" that already has 2 E's: skips E on belt
    - Inserter facing an assembler with no recipe: picks up any letter
    - Inserter not facing an assembler (belt-to-belt): picks up any item

    Belt-to-machine transfers:
    - Pickup from belt, deposit into assembler: calls assembler.depositLetter()
    - Pickup from assembler.outputItem, deposit onto belt: moves WordItem to belt

    Long-arm inserter:
    - Picks up from belt at pickupTile() (1 or 2 tiles away based on pickupRange)
    - Deposits at dropoffTile() (always 2 tiles away)

    Edge cases:
    - No item at pickup tile: inserter does nothing, timer still increments
    - Inserter with heldItem skips pickup phase, only tries deposit
    - Multiple inserters feeding same assembler: each checks neededCount independently per tick
  </behavior>
  <action>
    Create InserterSystem as a pure TS class. No Phaser imports.

    Constructor takes no args. Has `addInserter(inserter)`, `removeInserter(inserter)`, `getInserterAt(x, y)`, `getAllInserters()`.

    `tick(beltSystem: BeltSystem, buildingSystem: BuildingSystem): void` iterates all inserters:
    1. Increment ticksSinceLastTransfer
    2. If < INSERTER_TRANSFER_INTERVAL, skip
    3. If heldItem is not null, try deposit (tryDeposit)
    4. If heldItem is null, try pickup (tryPickup)

    tryPickup logic:
    - Get source at pickupTile(): could be belt (beltSystem.getBeltAt) or assembler (buildingSystem.getAt with type check)
    - If source is belt with item: check smart filter (if dropoff target is assembler with recipe, only grab if assembler.neededCount(item.letter) > 0)
    - If source is assembler with outputItem: grab the WordItem
    - On successful pickup: set heldItem, clear source, reset ticksSinceLastTransfer = 0

    tryDeposit logic:
    - Get target at dropoffTile(): could be belt or assembler
    - If target is belt: only deposit if belt.item is null
    - If target is assembler: call assembler.depositLetter(). If returns false (not needed), hold item.
    - On successful deposit: clear heldItem, reset ticksSinceLastTransfer = 0

    Handle both Inserter and LongArmInserter — they share the same interface pattern (pickupTile/dropoffTile).

    Write tests first (RED), then implement (GREEN).
  </action>
  <verify>
    <automated>bun test InserterSystem</automated>
  </verify>
  <done>InserterSystem handles all transfer scenarios with smart filter. All tests pass including edge cases.</done>
</task>

<task type="auto" tdd="true">
  <name>Task 2: AssemblerSystem with ingredient collection and assembly timer</name>
  <files>
    src/game/systems/AssemblerSystem.ts,
    src/game/systems/AssemblerSystem.test.ts
  </files>
  <behavior>
    Assembly lifecycle:
    - Assembler with recipe "SET" and all 3 letters deposited: isReadyToAssemble() returns true
    - On next tick after ready: assemblyTicksRemaining set to ASSEMBLY_TICKS_BASE + word.length * ASSEMBLY_TICKS_PER_LETTER
    - Each subsequent tick: assemblyTicksRemaining decrements by 1
    - When assemblyTicksRemaining reaches 0: outputItem set to new WordItem(word, assemblerCenterTile)
    - OutputItem stays until an inserter pulls it (tested in InserterSystem tests)
    - After outputItem is pulled (becomes null): assembler.reset() clears ingredients, ready for next word

    Edge cases:
    - No recipe assigned: tick does nothing
    - Partial ingredients: tick does nothing (no timer starts)
    - Timer already running (assemblyTicksRemaining > 0) and outputItem is null: continue decrementing
    - assemblyTicksRemaining already at 0 and outputItem exists: do not re-create outputItem
    - Multiple assemblers: each processed independently
  </behavior>
  <action>
    Create AssemblerSystem as a pure TS class. No Phaser imports.

    Has `addAssembler(assembler)`, `removeAssembler(assembler)`, `getAssemblerAt(x, y)`, `getAllAssemblers()`.

    `tick(): void` iterates all assemblers:
    1. If assembler.recipe is null, skip
    2. If assembler.outputItem is not null, skip (waiting for output to be pulled)
    3. If assembler.assemblyTicksRemaining > 0, decrement by 1. If now 0, create WordItem and set assembler.outputItem
    4. If assembler.assemblyTicksRemaining === 0 and assembler.isReadyToAssemble(), start timer: set assemblyTicksRemaining = ASSEMBLY_TICKS_BASE + recipe.word.length * ASSEMBLY_TICKS_PER_LETTER

    The assembler center tile (for WordItem position) is calculated as tileX + 1, tileY + 1 (center of 3x3).

    `getAssemblerAt(x, y)` checks if any assembler's occupiedTiles() contains (x, y) — used by InserterSystem to identify assembler targets. Cache or iterate; the number of assemblers is small enough that iteration is fine.

    Write tests first (RED), then implement (GREEN).
  </action>

  </verify>
