<task type="auto" tdd="true">
  <name>Task 1: SplitterSystem with auto-mode detection</name>
  <files>
    src/game/systems/SplitterSystem.ts,
    src/game/systems/SplitterSystem.test.ts
  </files>
  <behavior>
    Splitter mode (1-in / 2-out):
    - Belt feeding into input side has item → item moves to next output belt, alternating left/right
    - If target output belt is full, try the other output; if both full, item stays on input belt
    - Alternation state persists across ticks (strict 1:1)

    Merger mode (2-in / 1-out):
    - Two input belts, one output belt: items from both inputs merge onto output
    - If both inputs have items, alternate which input gets priority
    - If output belt is full, neither input advances

    Balancer mode (2-in / 2-out):
    - Cross-routes: left-in goes to right-out, right-in goes to left-out (when possible)
    - Falls back to same-side if cross-route output is full

    Mode detection:
    - Count input and output adjacent belts each tick to determine mode
    - Input belt = belt adjacent to splitter whose direction points INTO the splitter
    - Output belt = belt adjacent to splitter on the output side of the splitter's direction

    Edge cases:
    - No adjacent belts: splitter does nothing
    - Splitter with only 1 input and 1 output: pass-through (no alternation)
  </behavior>
  <action>
    Create SplitterSystem as a pure TS class. No Phaser imports.

    Has `addSplitter(splitter)`, `removeSplitter(splitter)`, `getAllSplitters()`.

    `tick(beltSystem: BeltSystem): void` iterates all splitters:
    1. Determine input side and output side based on splitter.direction (input = opposite of direction, output = direction side)
    2. Find adjacent belts on input side and output side using beltSystem.getBeltAt()
    3. For each input belt with an item, try to route it to an output belt
    4. Use splitter.nextOutputSide() for alternation when multiple outputs available

    The splitter does NOT register in BeltSystem — it sits between belt chains. BeltSystem sees the belt before a splitter as "end of chain" (getNextBelt returns null). SplitterSystem explicitly pulls from input belts and pushes to output belts.

    Key detail: splitter occupies 2 tiles perpendicular to flow direction. The "left" and "right" output tiles are relative to the splitter's direction. For a splitter facing East: input belt at west-adjacent tile, left output at the north-side output tile, right output at the south-side output tile.

    Write tests first (RED), then implement (GREEN).
  </action>
  <verify>
    <automated>bun test SplitterSystem</automated>
  </verify>
  <done>SplitterSystem handles split, merge, and balance modes with correct alternation. All tests pass.</done>

<task type="auto" tdd="true">
  <name>Task 2: UndergroundBeltSystem with tick-delayed buffer</name>
  <files>
    src/game/systems/UndergroundBeltSystem.ts,
    src/game/systems/UndergroundBeltSystem.test.ts
  </files>
  <behavior>
    Item transit:
    - Belt adjacent to entry (feeding in) has item → item moves into entry.buffer with ticksRemaining = entry.tunnelLength
    - Each tick: all buffer items decrement ticksRemaining by 1
    - When ticksRemaining reaches 0: item moves to belt adjacent to exit (output side)
    - If exit belt is occupied: item stays in buffer at ticksRemaining=0 until space opens

    Pairing:
    - addPair(entry, exit) links them bidirectionally
    - removePair clears both pair references
    - Entry without paired exit: items entering buffer are stuck (no destination)

    Multiple items in buffer:
    - Items maintain order (FIFO) — first in, first out at exit
    - Multiple items can be in transit simultaneously with different ticksRemaining values

    Edge cases:
    - No input belt adjacent to entry: nothing happens
    - Entry buffer item at ticksRemaining=0 but no exit belt: item stays in buffer
    - Entry without pair: do not accept items from input belt
  </behavior>
  <action>
    Create UndergroundBeltSystem as a pure TS class. No Phaser imports.

    Has `addPair(entry, exit)`, `removePair(entry)`, `getAllEntries()`, `getAllExits()`.

    `tick(beltSystem: BeltSystem): void`:
    1. For each entry with a paired exit:
       a. Check input belt (belt adjacent on the opposite side of entry.direction). If it has an item and entry.pair exists, move item to entry.buffer with ticksRemaining = entry.tunnelLength
       b. Decrement all buffer items' ticksRemaining by 1 (but not below 0)
       c. Check if first buffer item (FIFO) has ticksRemaining === 0: if exit output belt (belt adjacent on the direction side of exit) is empty, move item to that belt, remove from buffer

    The input belt for an entry is the belt adjacent on the side OPPOSITE to the entry's direction (items flow into the entry from behind). The output belt for an exit is the belt adjacent on the direction side (items flow out in that direction).

    Note: Underground belt tiles do NOT block building placement on tiles above them. This is handled by the fact that underground "tunnel" tiles between entry and exit are not registered in BuildingSystem — only the entry and exit 1x1 tiles are registered. Any tile between them is free for building placement.

    Write tests first (RED), then implement (GREEN).
  </action>

    <automated>bun test UndergroundBeltSystem</automated>
  </verify>
  <done>UndergroundBeltSystem correctly delays items through the tunnel buffer and delivers them at the exit. FIFO ordering maintained. All tests pass.</done>
