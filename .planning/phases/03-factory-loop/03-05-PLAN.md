  <name>Task 1: Renderers for all new building types</name>
  <files>

  <action>
    Create 4 new renderer classes following the established pattern: constructor(scene, system, debugOverlay), addX/removeX, update(), destroy().

    **InserterRenderer:**
    - Each inserter: draw base circle at tile center (DEPTH_BUILDINGS) and arm line from base toward pickup side
    - `update(alpha)`: interpolate arm position between pickup and dropoff based on inserter state:
      - idle: arm points toward pickup side
      - holding item and swinging to drop: lerp rotation from pickup angle to dropoff angle using alpha
      - Show held letter on the arm tip during swing (small colored square with letter text)
    - Arm swing uses the inserter's `ticksSinceLastTransfer` progress and `heldItem` state
    - Use `DEPTH_INSERTER_ARM` for arm (between buildings and items layer)
    - `addInserter(inserter)`: create Phaser graphics objects
    - `removeInserter(inserter)`: destroy graphics objects

    **AssemblerRenderer:**
    - Draw 3x3 tiled block at assembler position (DEPTH_BUILDINGS) with distinct color (e.g., muted blue-grey to contrast against green ground)
    - Show recipe word text centered on the assembler (if recipe assigned)
    - Progress display: draw small letter slots along the bottom edge of the assembler. Each slot shows the needed letter. Filled slots are bright, unfilled are dim. This gives visual feedback on collection progress.
    - When assembling (assemblyTicksRemaining > 0): show a small progress bar or pulsing effect
    - When outputItem exists: show word text highlighted, ready for pickup
    - `addAssembler(assembler)`: create container with background, text, slots
    - `removeAssembler(assembler)`: destroy container
    - `update()`: refresh progress display based on assembler state

    **Recipe Panel (part of AssemblerRenderer):**
    - When GameScene calls `showRecipePanel(assembler)`, create a Phaser Container on the UI camera:
      - Semi-transparent dark background panel
      - Title: "Select Recipe"
      - List of available recipes from WORD_DICTIONARY, grouped by tier
      - Each recipe shows: word, required letters, unique letter count
      - Clicking a recipe assigns it to the assembler and closes panel
      - Clicking outside panel closes it
    - `hideRecipePanel()`: destroys the panel container
    - Panel is rendered on UI camera (setScrollFactor(0), ignored by main camera)

    **SplitterRenderer:**
    - Draw 1x2 block at splitter position (DEPTH_BUILDINGS) with arrows showing split direction
    - Static render (no animation needed — items appear on output belts via BeltRenderer)
    - `addSplitter(splitter)`: create graphics
    - `removeSplitter(splitter)`: destroy graphics

    **UndergroundBeltRenderer:**
    - Draw entry tile with downward arrow indicator (item going underground)
    - Draw exit tile with upward arrow indicator (item emerging)
    - Optional: dashed line between entry and exit showing tunnel path (at low alpha)
    - `addPair(entry, exit)`: create graphics for both
    - `removePair(entry)`: destroy graphics for both

    **ItemRenderer extension:**
    - Update syncItems() to handle BeltItem union (LetterItem | WordItem)
    - For LetterItem: unchanged (colored square with single letter)
    - For WordItem: wider rectangle (ITEM_SIZE * word.length * 0.6) with word text, using same color palette as letters
    - Type check: `if ('word' in item)` to distinguish WordItem from LetterItem

    All renderers must register their game objects with debugOverlay.ignoreOnUiCamera() to prevent double-rendering on the UI camera, following the existing pattern.
  </action>

  <action>
    **GameScene.create() additions:**
    1. Instantiate new systems:
       ```
       this.inserterSystem = new InserterSystem();
       this.assemblerSystem = new AssemblerSystem();
       this.splitterSystem = new SplitterSystem();
       this.undergroundBeltSystem = new UndergroundBeltSystem();
       ```
    2. Instantiate new renderers:
       ```
       this.inserterRenderer = new InserterRenderer(this, this.inserterSystem, this.debugOverlay);
       this.assemblerRenderer = new AssemblerRenderer(this, this.assemblerSystem, this.debugOverlay);
       this.splitterRenderer = new SplitterRenderer(this, this.splitterSystem, this.debugOverlay);
       this.undergroundBeltRenderer = new UndergroundBeltRenderer(this, this.undergroundBeltSystem, this.debugOverlay);
       ```

    **GameScene.update() tick callback additions:**
    Add to tick callback (after quarrySystem.tick() and beltSystem.tick()):
    ```
    this.splitterSystem.tick(this.beltSystem);
    this.undergroundBeltSystem.tick(this.beltSystem);
    this.inserterSystem.tick(this.beltSystem, this.buildingSystem);
    this.assemblerSystem.tick();
    this.itemRenderer.syncItems();
    ```
    Order matters: belts move items first, then splitters/underground route them, then inserters transfer between belts/machines, then assemblers process. syncItems at end.

    **GameScene.update() per-frame additions:**
    ```
    this.inserterRenderer.update(this.tickEngine.alpha);
    this.assemblerRenderer.update();
    ```

    **Keyboard hotkey changes:**
    Replace current hotkey bindings with Phase 3 layout:
    - keydown-ONE: Belt (unchanged)
    - keydown-TWO: Splitter
    - keydown-THREE: Underground
    - keydown-FOUR: Inserter
    - keydown-FIVE: Assembler
    - keydown-X: Demolish
    - keydown-DELETE: Demolish (alternate)
    - keydown-ESC: deselect tool + clear pendingUndergroundEntry
    - keydown-R: placementSystem.rotateDirection() (only when a rotation-supporting tool is active)

    **Pointer handler changes:**
    Update pointerdown handler:
    - If currentTool === ToolType.None: check getConfigTarget() for click-to-configure
      - If target is assembler: open recipe panel via assemblerRenderer.showRecipePanel(assembler)
      - If target is inserter (standard or long-arm): show info or toggle long-arm range
    - If currentTool === ToolType.Underground:
      - If no pendingEntry: validate and place entry (add to BuildingSystem + UndergroundBeltSystem), set pendingEntry
      - If pendingEntry exists: validate exit position, place exit, link pair, clear pendingEntry
      - If exit invalid: do nothing (keep pendingEntry)
    - If currentTool is Inserter/Assembler/Splitter: single-click placement
      - Create entity with current placementDirection
      - Validate footprint via isValidFootprint
      - Add to BuildingSystem + respective system + renderer
    - If currentTool is Belt/Demolish: existing drag behavior (unchanged)

    Update pointerup handler:
    - After placing a belt, also check if belt connects to splitter/underground — refresh as needed
    - After demolish, also remove from inserterSystem/assemblerSystem/splitterSystem/undergroundBeltSystem if the demolished building matches

    **Demolish handler updates:**
    When demolishing a tile, check building type and remove from the correct system:
    - "inserter" or "long-arm-inserter": remove from inserterSystem + inserterRenderer
    - "assembler": remove from assemblerSystem + assemblerRenderer
    - "splitter": remove from splitterSystem + splitterRenderer
    - "underground-entry": remove pair from undergroundBeltSystem + undergroundBeltRenderer
    - "underground-exit": same (remove via the pair's entry)
    - "belt": existing behavior

    **BootScene:**
    No new sprite assets to load for Phase 3 — all rendering uses Phaser Graphics/Text primitives (same as existing renderers). If any sprite keys are needed by renderer constructors, add placeholder programmatic textures here.
